# HDGP 规则包签名与验证（草案）

> 本规范描述规则包（Policy Bundle）的**签名与验证**流程占位与未来方向，与《HDGP 自身伦理框架基线》§7.2（防静默篡改）及 `GOVERNANCE.md` 多签要求一致。  
> 当前实现中签名为**占位**，Engine 不强制校验；后续 Alpha 阶段可实现真实校验与拒绝加载未签名/校验失败的规则包。

---

## 一、目标

- 确保规则包在分发与加载过程中**不被静默篡改**。  
- 任何对规则包内容的修改，必须经**人类治理流程**与**多签**后以新版本发布；单点无法私自改写。  
- Engine 加载规则包时，可验证签名与期望清单一致，否则拒绝加载或记录 `integrity_events` 并告警。

---

## 二、规则包签名字段（占位）

规则包元数据中预留以下字段（与 `HDGP_CORE_MAPPING_SPEC.md` 2.3 及 `internal/engine/types.go` 中 `PolicySignature` 一致）：

| 字段 | 说明 |
|------|------|
| `signature.key_id` | 公钥或证书标识，用于验证时定位信任根中的公钥。 |
| `signature.value` | 对规则包规范序列化内容（如 `bundle_id` + `spec_version` + `rules` 的确定性序列化）的签名值，Base64 或十六进制编码。 |
| `signature.algorithm` | 签名算法，如 `SHA256-RSA`、`Ed25519` 等。 |

当前 MVP 中，Engine 不从外部文件加载规则包，策略与规则包 ID 在代码中写死；上述字段仅在 Meta 与类型定义中预留，供未来“从文件/配置加载规则包”时使用。

---

## 三、验证流程（未来）

1. **加载前**：Engine 读取规则包（文件或配置）及附带的签名。  
2. **取公钥**：根据 `signature.key_id` 从**期望版本/签名清单**或本地信任根中取对应公钥。  
3. **验签**：对规则包规范序列化结果使用公钥验证 `signature.value`；若算法不支持或验签失败，则：  
   - **拒绝加载**该规则包，并记录 `integrity_events`（如 `signature_failed`）；  
   - 可选：向上报告或对外告警（见伦理基线 §7.3）。  
4. **多签**：若治理要求多签，可约定规则包需携带多个 `signature` 条目，Engine 校验时要求至少 N 个签名有效。

---

## 四、期望版本/签名清单来源

- **配置**：部署时在配置文件中指定“官方”规则包版本列表及对应公钥或证书。  
- **信任根**：项目或部署方维护一个**期望版本清单**（如 JSON/YAML），列出 `bundle_id`、`spec_version`、允许的 `key_id` 或签名摘要；Engine 启动时或定期拉取/读取该清单，与本地加载的规则包比对。  
- **来源安全**：清单本身可通过 HTTPS、HashiCorp Vault、或离线交付等方式保证完整性；清单的变更应经治理流程。

---

## 五、与现有实现的关系

- `internal/engine/types.go` 中已定义 `PolicySignature` 与 `MetaPolicy.Signature`；`EvaluateResponse` 与审计日志中已预留 `integrity_events`。  
- 当前 Engine **不**从外部加载规则包，因此无签名校验逻辑；后续在实现“从文件/远程加载规则包”时，须在本规范基础上实现校验并在校验失败时拒绝加载或告警。

本文档为草案，随实现与治理演进可增补具体算法、清单格式与多签策略。
