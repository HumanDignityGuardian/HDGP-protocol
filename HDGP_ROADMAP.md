## HDGP 项目技术路线与里程碑（草案）

本文件评估：从当前状态出发，将 HDGP 从“设计与文档”推进到“可稳定运行、可对外（组织）开放的技术系统”的大致难度与时间，并给出阶段性技术实现方案与里程碑。  
所有时间估算均以**全职 1–2 名工程师 + 少量临时协作者**为参考，可根据实际人力线性缩放。

---

## 一、总体难度与时间评估

### 1. 难度评估

- **技术复杂度：中–偏高**
  - 涉及：API 网关、策略引擎、规则系统、日志与审计、与大模型/工具的集成、不确定性评估等；  
  - 但每个单点都可以先做 MVP，再逐步增强。
- **设计复杂度：高**
  - 需要保证伦理框架、规范与实现三者的一致性；  
  - 需要在“灵活可扩展”与“硬约束不可绕过”之间取得平衡。
- **生态与治理复杂度：中–高**
  - 标准化接口、合规测试、社区参与与治理机制都需要时间沉淀。

### 2. 时间评估（从 0 开发到对组织开放的首个稳定版）

在“1–2 名主力开发 + 兼职参与”的假设下：

- **阶段 0：准备期（0.5 个月）**  
  - 整理规范、架构定稿、小范围 PoC 选型。
- **阶段 1：最小可用版（MVP）（1.5–2 个月）**  
  - 单模型、单场景（如问答/决策建议）的 HDGP 网关 + 基础策略引擎 + 日志；  
  - 能在受控环境下真实跑起来。
- **阶段 2：Alpha 内测版（2–3 个月）**  
  - 引入 Meta/Skill/Workflow/Engine 全栈架构；  
  - 增强策略引擎、不确定性熔断机制、合规测试套件；  
  - 内部或少数合作方试用，验证可用性与安全边界。
- **阶段 3：Beta 组织开放版（3–4 个月）**  
  - 多模型、多场景支持；  
  - Web 控制台与观测性；  
  - 合规测试流程与文档完善，允许外部组织接入试点。

**合计粗略估算：7–9 个月** 可达“可对组织开放试点”的 Beta 水平。  
若人力更充足 / 有现成基础设施，可适当缩短；若为单人兼职推进，则需相应放大。

## 二、分阶段技术实现方案

### 阶段 0：准备与 PoC（约 0.5 个月）

**目标**：从技术和产品角度把核心选择锁定在一个可执行的最小集合上。

- **0.1 技术选型与基础设施**
  - 语言/运行时：  
    - 后端：Go / Rust / TypeScript(Node) 三选一（偏向 Go 或 Rust）；  
    - 脚本与快速实验：Python。
  - 存储：  
    - 配置/规则/审计索引：PostgreSQL；  
    - 日志与事件：文件 + 将来可接 Kafka/日志平台。
  - 模型接入：  
    - 先选 1–2 个主流 LLM（如 OpenAI / 本地 LLaMA 系）；  
    - 通过统一适配层封装。

- **0.2 PoC 验证**
  - 写一个极简“HDGP 网关”原型：  
    - `/hdgp/v1/chat` → 转发到 LLM → 简单规则：禁止直接下最终决策、标记置信度、输出免责声明；  
    - 记录结构化日志。  
  - 验证：  
    - 性能是否可接受；  
    - 与预想的 Meta/Skill/Workflow/Engine 架构是否兼容。

**里程碑 M0**：  
完成 PoC，确认技术栈与总体架构方向；形成技术决策文档（可附在 `spec/` 下）。

---

### 阶段 1：MVP 网关与基础规则（约 1.5–2 个月）

**目标**：实现一个可供“单一团队内部使用”的 HDGP 网关 MVP。

- **1.1 基础网关与 API**
  - 实现统一入口：  
    - `/hdgp/v1/chat`（文本对话）；  
    - `/hdgp/v1/decision`（决策建议）；  
  - 请求/响应结构中包含：  
    - 用户/会话 ID（匿名化）；  
    - 场景标签（手动或简单自动分类）；  
    - HDGP 版本号与规则包 ID。

- **1.2 策略引擎与只读规则包 v0**
  - 定义简单策略 DSL 或接入 OPA/Rego：  
    - 规则示例：  
      - 禁止使用命令式语气给出唯一决策；  
      - 在高风险关键词出现时，附加免责声明并建议咨询专业人士。  
  - 规则包打包为独立工件：  
    - 带有版本号与元数据，但暂不做签名（后续阶段补）。

- **1.3 基础“不确定性熔断”**
  - 利用模型置信度 / 多次采样差异度，做一个简单的不确定性评分；  
  - 当评分低于阈值且场景为高风险时：  
    - 拒绝给出明确建议；  
    - 给出“无法负责任地回答”的提示 + 建议后续行动。

- **1.4 审计日志与简单查询工具**
  - 每次请求记录：输入摘要、输出摘要、触发规则、熔断与否；  
  - 提供 CLI 或简单 HTTP 端点查看最近日志。

**里程碑 M1（MVP）**：  
在受控环境中，HDGP 网关可以稳定运行，为某个内部应用提供“尊严保护层”，并且出现明显高风险请求时能按预期熔断或降级回答。

---

### 阶段 2：Alpha 内测版（约 2–3 个月）

**目标**：引入完整的 Meta + Skill + Workflow + Engine 架构，并强化安全性和规则管理。

- **2.1 明确 Meta / Skill / Workflow / Engine 的数据结构与接口**
  - 在 `spec/` 下定义：  
    - `Meta`：用户类型、场景、风险等级、价值偏好、规范版本等字段；  
    - `Skill`：能力描述、输入/输出模式、约束（允许的场景等）；  
    - `Workflow`：按节点编排调用 Skills 与 Engine；  
    - `Engine`：统一的规则评估接口（如 `evaluate(meta, intent, result)`）。

- **2.2 引擎（Engine）实现 v1**
  - 加载只读规则包（从 `policies/`）；  
  - 对所有请求与技能调用进行：  
    - 前置评估（意图合法性）；  
    - 后置评估（输出是否合规）；  
    - 统一触发熔断与人工介入提示。  
  - 实现与业务代码解耦：通过 SDK 或 HTTP/gRPC 调用 Engine。

- **2.3 策略与规则增强**
  - 将白皮书中的几条核心原则转化为可执行规则：  
    - 不代替人类做终极决策；  
    - 不利用心理弱点进行操纵；  
    - 不提供单一宿命论式未来叙事；  
    - 保护非理性与多样性。  
  - 支持按场景加载不同策略 profile（如医疗/金融/教育）。

- **2.4 合规测试套件 v1**
  - 建立 `conformance-tests/`：  
    - 设计若干关键测试用例（JSON/YAML），覆盖高风险场景；  
    - 实现自动化测试 runner，对 HDGP 实例发起请求并验证返回行为。

- **2.5 内测集成与反馈**
  - 选 1–2 个真实项目做内测（可为你自己的其他项目）；  
  - 收集：  
    - 熔断率是否过高/过低；  
    - 对话体验与提示文案是否可接受；  
    - 日志与审计数据是否足够支撑复盘。

**里程碑 M2（Alpha）**：  
HDGP 已经具备完整的架构骨架，核心规则通过 Engine 统一执行，合规测试可自动跑通；在少量早期伙伴或内部项目中，可以作为“副驾驶防火墙”稳定运行。

---

### 阶段 3：Beta 组织开放版（约 3–4 个月）

**目标**：增强多场景、多模型支持，引入基础 Web 控制台与更完善的治理/观测能力，达到可向外部组织开放试点的水平。

- **3.1 多模型与多后端支持**
  - 在 `adapters/` 中支持多种 LLM 与工具后端；  
  - 支持为不同租户/项目配置不同的模型与限额；  
  - 引入基础熔断与超时处理机制（避免下游故障导致整体不可用）。

- **3.2 Web 控制台 v1**
  - 提供基础的管理界面（可先用简单的前端框架）：  
    - 查看当前规则包版本与 Engine 状态；  
    - 配置场景 profile 与策略组合（在可配置范围内）；  
    - 查看简化版审计日志与统计数据。

- **3.3 规则包签名与版本管理 v1**
  - 实现规则包（Policy Bundle）的签名与验证流程：  
    - 使用固定公钥验证规则包的完整性与来源；  
    - 在 Engine 启动与规则加载时强制校验签名。  
  - 建立规则包版本清单文档（对应 HDGP 规范版本）。

- **3.4 观测性与告警**
  - 统计与暴露关键指标：  
    - 请求量、熔断次数与比例；  
    - 不同场景的风险等级分布；  
    - 规则命中统计。  
  - 为将来对接 Prometheus/Grafana 等监控工具预留接口。

- **3.5 文档与对外接口**
  - 完成面向组织用户的文档：  
    - 如何部署 HDGP 网关；  
    - 如何接入现有应用；  
    - 如何理解熔断与错误信息；  
    - 合规测试如何运行与解读。  
  - 根据 `GOVERNANCE.md` 与 `HDGP_COMMERCIAL_PLAN.md`，准备对外试点协议与合作模式。

**里程碑 M3（Beta 组织开放版）**：  
HDGP 可以作为独立服务，对外提供稳定 API、基础管理界面与可复现的合规测试流程。  
适合邀请少量外部组织参与试点，并根据反馈继续优化。

## 三、风险与不确定性

- **需求与边界漂移风险**：  
  - HDGP 本身仍在概念演进期，可能在实践中发现某些条款需要调整；  
  - 需要在规范层预留版本化与修订流程，并通过 Engine 与规则包版本解耦实现。

- **性能与成本风险**：  
  - 在大规模高并发场景下，熔断、审计与多次调用模型会显著增加延迟与成本；  
  - 需要在后期引入缓存、降级策略与可配置的安全等级（不同组织可权衡“速度 vs 安全”）。

- **用户体验风险**：  
  - 过于频繁的拒绝与熔断可能导致用户挫败感；  
  - 需要通过精细化策略与文案设计，在“守护尊严”与“可用性”之间寻找平衡。

- **生态与治理风险**：  
  - 若缺乏足够参与者与真实场景反馈，规范可能会“纸上谈兵”；  
  - 需要主动寻找合作伙伴与试点场景，同时保证早期版本不被过度夸大能力。

---

## 四、总结

在当前基础（白皮书 + 项目骨架文档 + 开源与商业化框架）之上，HDGP 要走到“可对组织开放”的稳定 Beta，大致需要 **7–9 个月的持续工程投入**。  
建议按照上述阶段推进，每个阶段都以“可以被真实使用的小闭环”为目标，确保协议不只是概念，而是不断在现实场景中被检验和修正。

## 五、当前阶段完成总结（MVP 收尾与国际化）

截至本阶段，以下工作已完成，对应**阶段 1（MVP）**的收尾与对外展示准备。

### 5.1 核心实现

- **Engine 与 API**：Go 实现的 HDGP Engine 已提供 `/hdgp/v1/evaluate`、`/hdgp/v1/chat`、`/hdgp/v1/audit`、`/hdgp/v1/appeal`、`/hdgp/v1/status`，规则判定与审计、上诉为内存实现，满足本地 PoC 与合规测试。
- **核心规则**：4 条规则已实现并纳入合规测试——唯一正确决策（R-P-A2-01-01）、虚假承诺（R-P-A3-01-01）、利用心理弱点操纵（R-P-A2-02-01）、去人性化表述（R-P-A2-03-01）；规则冲突时的裁决顺序已在规范与实现中体现。
- **合规测试**：`conformance-tests/cases/` 下 7 个用例（含 evaluate 与 status），`go run ./cmd/hdgp-conftest` 可一键跑通；脚本 `scripts/run-conftest.ps1` 便于本地验证。

### 5.2 文档与国际化

- **README / CONTRIBUTING**：采用「英文在前、中文在后」结构，便于全球开发者阅读；英文部分包含完整的 Core Vision、Repository Layout、Open Framework、Who Is This For、Status & Roadmap、Kernel & Audit、Running & Conformance Tests、**Minimal Integration Example (curl)**、How to Contribute、Ethics & Disclaimer、Maintainer & License。
- **index.html**：顶部提供 EN/中文 切换，默认英文展示；全站关键区块（Hero、Supreme Directive、Who Is HDGP For、MVP Demo、Minimal API Example、三层结构、Kernel & Audit、合作模式与当前进展、未来愿景、页脚）均完成中英双语映射，无遗漏纯中文段落。
- **最小接入示例**：README 与 index 均提供可直接调用的 `/hdgp/v1/evaluate` curl 示例（含 `meta`、`subject`、`input`、`candidate`），方便试点项目集成。

### 5.3 内核与治理

- **内核自查与伦理**：`spec/HDGP_KERNEL_CHECKLIST.md` 覆盖文档一致性、防挟持能力、治理流程与测试设计；`spec/HDGP_ETHICS_BASELINE.md` 明确 §7 防挟持（只读内核、签名与自报警）、§8 宪章/伦理变更多层治理；`GOVERNANCE.md` 与 `docs/CHIP_PROCESS.md` 规定角色、决策流程与 CHIP 流程。
- **技术债与安全**：已单独成文 **`spec/HDGP_TECHNICAL_DEBT_AND_SECURITY_CHECKLIST.md`**，列出 Alpha 及后续阶段的技术债与安全加固项，供迭代与审计使用。

### 5.4 下一步

- 进入 **阶段 2（Alpha）** 后，按 `HDGP_NEXT_DEVELOPMENT_PLAN.md` 与 `spec/HDGP_TECHNICAL_DEBT_AND_SECURITY_CHECKLIST.md` 推进：规则与行为目录扩展、规则包签名与完整性校验实现、审计持久化与访问控制等。
